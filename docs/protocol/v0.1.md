> **Changes since v0.1.34 (2026-06-22) — escalation push-gating: clarify is stakes-gated + subject-configurable (roadmap Q1).** No wire change; a reference-implementation policy refinement on which consult signals WAKE the subject. v0.1.34 pushed on both blocking signals unconditionally; the two are not equal. **`escalate`** ("hold for a live human go/no-go") stays unconditional — it always pushes. **`clarify`** ("I need more info before acting") is the agent's default reflex when uncertain, so it fires often, including on low-stakes calls — pushing every one would cry wolf. So a `clarify` now wakes the subject ONLY when the action is also **consequential: `reversible === false` OR `magnitude === 'high'`** (`shouldEscalate(signal, action_descriptor, policy)` / `isConsequential`). Below that bar `phaedo_escalate` returns `escalated:false` with `reason:"clarify_below_threshold"` and tells the autonomous agent to resolve the clarify in its own flow (ask in-conversation / state a reasonable assumption) — it surfaces in the deferred review queue, no interruption. Unknown stakes (no descriptor) reads as low-stakes for clarify; the agent can always raise an explicit `escalate`. **The knob:** the stakes gate is a DEFAULT — a subject can override how `clarify` pushes, globally and per-domain, via an **escalation push policy** that rides the vault (portable: `phaedo_fingerprint.escalation_policy`, syncs across devices) and/or a local file (`PHAEDO_ESCALATION_POLICY`, file overrides vault), the same dual home as consult policies. Shape: `{ "clarify": { "default": "always"|"consequential"|"never", "by_domain": { "<domain>": "<mode>" } } }`; a per-domain mode (substring match) beats the default. `escalate` is deliberately NOT configurable (suppressing a "needs-a-human" wake would strand a blocking decision). Reference: `mcp/escalation.js` (`shouldEscalate`/`isConsequential`/`loadEscalationPolicy`/`resolveClarifyMode`), `mcp/server.js` (`phaedo_escalate`), `mcp/escalation-policy.sample.json`; locked by `mcp/test-escalation.mjs`. `phaedo_protocol_version` stays `0.1`.

> **Changes since v0.1.28 (2026-06-19) — origin-aware resolution: the act-as-me channel goes live (§4.2 ⇄ §10).** No wire change; additive. ## 1. Introduction

The Phaedo Protocol defines how a portable cognitive fingerprint is represented, encrypted, signed, transported, and consulted by AI sessions and agents.

The protocol is a wire and storage format plus a small set of consultation interfaces. It is not the algorithm that produces a fingerprint, and it is not the user interface that manages one. The protocol exists so that a fingerprint produced by the Phaedo reference implementation and a fingerprint produced by any conforming third party tool are interoperable. A consuming AI session, agent, or injection adapter that conforms to this protocol can read either one without modification.

> **Direction (2026-06-09): consultation-led.** The protocol defines two consumption
> surfaces — **Injection** (Section 9, *push*: a projection inserted at session start so
> an assistant responds in the subject's voice) and **Consultation** (Section 10, *pull*:
> an agent queries *how the subject would decide* at a fork and gets a constrained signal,
> never content). As of this revision the project's center of gravity is **consultation**:
> injection has become commoditized (mainstream assistants ship auto-memory), while a
> portable, user-owned **decision-style oracle** an agent consults — bounded by §10.4 and
> backed by subject-authored standing policies (§10.5) — is the distinguishing primitive.
> This is a shift of *emphasis*, not of normative content: Sections 4–9 are unchanged and
> injection remains a fully supported conforming surface. The hero layers for consultation
> are **Decision and Risk** (§4.2.7) and **Big Picture** (§4.2.8). Rationale and the
> brief↔spec mapping: `docs/roadmap/consultation-reframe-2026-06-09.md`.

This document is v0.1, an internal working draft maintained alongside the Phase 0 prototype build. The first public release is v1.0, targeted for Phase 2. Between v0.1 and v1.0 the schema and envelope formats may change in incompatible ways. After v1.0, semantic versioning applies and any breaking change requires a major version bump.

Machine-readable **JSON Schemas** for the fingerprint (Section 4) and the consultation surfaces (request 10.2, response 10.3, policy 10.5) are published alongside this document in `spec/schemas/` and are the normative machine form; the JSON examples in this prose are illustrative (see Section 6 for the canonical form).

### 1.1 Goals

1. **Portability.** Artifacts produced on one device (fingerprint and user configuration) are consumable by any conforming session or agent on any platform, given the user's consent and key material.
2. **User control.** Ciphertext is the only form of any artifact that ever leaves the user's vault device. Phaedo (the company) and any third party intermediary hold ciphertext only.
3. **Interoperability.** Any conforming producer or consumer can be substituted without coordination.
4. **Verifiability.** Integrity and authenticity are checkable at every boundary an artifact crosses.
5. **Forward compatibility.** Schema versioning, optional fields, and graceful degradation rules let v0.x consumers handle v0.y producers (and vice versa) within the same major version.

### 1.2 Out of Scope (v0.1)

- The persona signal extraction algorithm (proprietary; produces fingerprint content but is not described by the protocol).
- The Stained Glass obfuscation algorithm (proprietary; the protocol describes only the interface flag and the obfuscated payload envelope, not the obfuscation method itself).
- Enterprise admin tooling, fingerprint sync service APIs, billing protocols.
- The persona benchmark suite.
- The cryptographic format for cross device handoff (e.g., QR code export). v0.1 specifies the integrity envelope; the transport encoding is informative, not normative.

### 1.3 Document Conventions

Keywords MUST, MUST NOT, SHOULD, SHOULD NOT, MAY, and OPTIONAL are used as defined in RFC 2119 and RFC 8174.

JSON examples in this document are presented for readability and are not canonical. Section 6 specifies the canonical form used for signing and integrity checks.

Field names use `snake_case`. Identifiers, version strings, and algorithm identifiers are case sensitive.

### 1.4 Name, Governance, and Conformance Mark

**Name.** This standard is **Anamnesis** — a portable, vendor-neutral format for a subject's cognitive fingerprint and the interfaces that consume it. The name is Plato's term for knowledge as *recollection* (the *Meno*): the soul carries latent knowledge and re-surfaces it — which is exactly what a portable persona does across sessions. *Phaedo* is the reference implementation and the standard's initial steward, not the owner of the format ("Phaedo is the first and best implementation of Anamnesis"). "Phaedo Protocol" is retained as an **alias during v0.x** so existing references and the `phaedo_*` wire identifiers keep working; it is not a second standard.

**Wire identifiers.** v0.x retains the established `phaedo_*` field names and the `phaedo://` resource scheme as the normative identifiers (renaming them now would churn every existing artifact for no functional gain). Neutral `anam_*` field names and an `anam://` scheme become normative at **v1.0**, which is already the declared breaking-change boundary (§1, Introduction). v0.x → v1.0 producers/consumers SHOULD accept both as aliases across that cutover.

**Governance.** The standard is published openly (Apache-2.0 for the schemas/reference code, CC-BY for the prose) with a public, numbered RFC process (`docs/protocol/proposals/`). The intended home is a **neutral repository under a single steward** (Phaedo) — open and separable, without the overhead of a foundation; a multi-stakeholder steering model is adopted only once a second independent implementer exists. The open/proprietary boundary is fixed in Appendix A and does not move after v1.0.

**Conformance mark.** Authority over the standard rests in the **conformance program**, not the name — the USB-IF model (nobody owns USB; the conformance program shapes it). An **"Anamnesis Compatible"** certification mark MAY be granted to any implementation that passes the published conformance suite for its claimed level (§11): the §12 test vectors (`spec/test-vectors.mjs`), `phaedo-validate` clean for any fingerprints it emits (`spec/validate-fingerprint.mjs`), and a published §11.3 conformance statement. The mark is claimed per level — e.g. "Anamnesis Level 0 Compatible" (consumes a projection) through "Anamnesis Level 2 Compatible" (full producer) — so a chat client and a vault app advertise honest, differentiated conformance. The mark and its grant criteria are the durable control point; the schema is not.

## 2. Terminology

**Artifact.** A protocol level data object belonging to a subject. v0.1 defines two artifact kinds: the fingerprint (Section 4) and the user configuration (Section 5). All artifacts share the canonical serialization, encryption envelope, and integrity machinery (Sections 6 through 8).

**Fingerprint.** A structured, layered representation of a subject's cognitive patterns, sufficient for an AI session to align outputs with the subject's voice, expertise, decision style, and context. The fingerprint is not a transcript and is not a memory of conversations. It is a model derived from observation.

**User Configuration.** A subject authored artifact paired with a fingerprint. Carries standing instructions and an example corpus. Authored content rather than inferred patterns. See Section 5.

**Subject.** The natural person whose patterns and authored content are represented in the artifacts. The subject controls the keys.

**Layer.** A named partition of the fingerprint. Each layer covers a distinct dimension (domain, temporal, decision, etc.). Layers are independently selectable for injection and independently encryptable.

**Envelope.** The encrypted container holding the canonical bytes of an artifact, plus integrity and metadata fields. The same envelope format is used for both fingerprint and user configuration.

**Vault Device.** The device that holds the user's master key material in a secure element or trusted execution environment. In the Phaedo reference implementation, the vault device is a phone.

**Working Device.** The device on which the user runs an AI session that consumes an injection. Often a laptop. The working device receives a session scoped derived key, never the master key.

**Injection Adapter.** Software that inserts artifact projections (selected fingerprint layers, standing instructions, retrieved examples) into an AI session as system prompt or context. Plaintext artifact material exists only for the lifetime of an injection; a paired working-device client MAY additionally keep an at-rest cache of derived artifact material, but only as ciphertext (§7.4).

**Consulting Agent.** An autonomous agent that queries the fingerprint at decision points before taking consequential actions on the subject's behalf.

**Producer.** Any conforming software that creates or updates an artifact. A producer may produce one or both artifact kinds; conformance profiles distinguish Fingerprint Producer from Configuration Producer (Section 11.1).

**Consumer.** Any conforming software that reads an artifact, including injection adapters and consulting agents.

**Conformance Profile.** A named subset of the protocol that an implementation claims to support. See Section 11.

## 3. Architecture Overview

The protocol is layered:

```
+--------------------------------------------------------------------+
|  Application: AI session, agent, injection adapter                 |
+--------------------------------------------------------------------+
|  Consultation API (Section 10)   Injection API (Section 9)         |
+--------------------------------------------------------------------+
|  Fingerprint (Section 4)         User Configuration (Section 5)    |
+--------------------------------------------------------------------+
|  Canonical serialization (Section 6)                               |
+--------------------------------------------------------------------+
|  Encryption envelope (Section 7)                                   |
+--------------------------------------------------------------------+
|  Integrity and signing (Section 8)                                 |
+--------------------------------------------------------------------+
|  Transport binding (MCP, stdio, REST, ...) — see binding specs     |
+--------------------------------------------------------------------+
```

The core protocol is transport-agnostic: Sections 9 and 10 define message shapes, not how messages travel. How those shapes are carried over a concrete transport is specified in separate **binding** documents. The MCP binding — the path by which MCP-capable AI clients and agent frameworks reach Phaedo, and the primary transport for desktop and agent integration — is specified in `mcp-binding.md`.

The protocol carries two data artifacts: the fingerprint (an inferred behavioral model) and the user configuration (authored content). Both belong to the same subject, share the same envelope and integrity machinery, and are injected together at session start.

A typical flow:

1. Producer observes subject behavior on the working device. Raw observation is processed locally and discarded. A fingerprint delta is produced.
2. The delta is encrypted under the subject's key and merged into the canonical fingerprint on the vault device.
3. The user authors standing instructions and curates an example corpus through the configuration surface. The user configuration is encrypted under the same key material.
4. At session time, the user's session on the working device requests an injection. The injection adapter retrieves the encrypted fingerprint and the encrypted user configuration, decrypts them within a session scoped boundary, selects relevant layers and instructions, and inserts a combined projection into the AI session.
5. After the session, the projection is destroyed. Only the encrypted fingerprint and configuration persist.
6. For agent consultation, the consulting agent issues a query through the consultation API. The fingerprint answers in a constrained shape (Section 10). The agent does not receive the full fingerprint or configuration.

## 4. Fingerprint Schema

### 4.1 Top Level Document

The canonical fingerprint document is a single JSON object:

```json
{
  "phaedo_protocol_version": "0.1",
  "fingerprint_id": "f1b9c4e2-7d3a-4e8b-9c0f-2a1b3c4d5e6f",
  "subject_id": "opaque_subject_identifier",
  "created_at": "2026-04-28T17:00:00Z",
  "updated_at": "2026-04-28T17:00:00Z",
  "persona_strength": 0.62,
  "schema_revision": 1,
  "layers": {
    "surface": { ... },
    "behavioral": { ... },
    "creative": { ... },
    "domain_and_expertise": { ... },
    "temporal_and_context": { ... },
    "collaboration_and_relationship": { ... },
    "decision_and_risk": { ... },
    "big_picture": { ... }
  },
  "extensions": {}
}
```

Field definitions:

| Field | Type | Required | Description |
|---|---|---|---|
| `phaedo_protocol_version` | string | yes | Protocol version this document conforms to. Format: `MAJOR.MINOR`. |
| `fingerprint_id` | string (UUID v4) | yes | Stable identifier for this fingerprint instance. |
| `subject_id` | string | yes | Opaque identifier for the subject. MUST NOT be a personally identifying value. |
| `created_at` | string (RFC 3339) | yes | UTC timestamp of fingerprint creation. |
| `updated_at` | string (RFC 3339) | yes | UTC timestamp of most recent layer update. |
| `persona_strength` | number | yes | Scalar in `[0.0, 1.0]`. Producer's confidence that the fingerprint is a useful match. |
| `schema_revision` | integer | yes | Integer revision counter. Incremented on any layer schema change within the same protocol version. |
| `layers` | object | yes | Map of layer name to layer document. See Section 4.2. |
| `consult_policies` | object | no | Optional consultation deference policies (Section 10.5) carried with the fingerprint so they travel across every transport. See `spec/schemas/consultation-policy.schema.json`. |
| `extensions` | object | no | Reserved for producer specific or experimental fields. Consumers MUST ignore unknown extension keys. |

### 4.2 Layers

The fingerprint is partitioned into eight named layers. Each layer is a JSON object. A producer MAY omit any layer. A consumer MUST tolerate any combination of present and absent layers.

Each layer document conforms to a common envelope:

```json
{
  "layer_version": "0.1",
  "last_updated": "2026-04-28T17:00:00Z",
  "confidence": 0.0,
  "signals": [],
  "summary": ""
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `layer_version` | string | yes | Layer schema version, independent of top level protocol version. |
| `last_updated` | string (RFC 3339) | yes | Most recent observation that updated this layer. |
| `confidence` | number | yes | Scalar in `[0.0, 1.0]`. Producer's confidence in this layer specifically. |
| `signals` | array | yes | List of structured signal objects. Schema per layer. |
| `summary` | string | yes | Short natural language summary of the layer for injection. v0.1 limit: 2000 characters. |

The eight layers are presented in canonical ordering. This is the order producers SHOULD use when serializing the `layers` object and the order consumers SHOULD use when constructing a default projection.

#### 4.2.1 Surface

Communication style, tone, formality, vocabulary, format preferences.

Example signals: preferred sentence length distribution, em dash avoidance, hedge frequency, emoji usage, salutation style, formatting tendencies (bullets vs prose, headers vs flat).

#### 4.2.2 Behavioral

How the subject works the material. Iteration patterns, editing behavior, what the subject accepts and rejects in AI output, where polish stops. The "how you work" signals, distinct from the "how you decide" signals in Decision and Risk (Section 4.2.7).

Example signals: accept rate on AI suggestions by category (technical, narrative, structural), edit depth distribution, drafts before declaring done, where the subject corrects vs leaves rough, preference for outline first vs draft first, frequency of self correction.

#### 4.2.3 Creative

Aesthetic sensibilities, mental models, how the subject reasons by analogy.

Example signals: analogy density, preferred metaphor domains, aesthetic stop conditions, recurring framing instincts, what counts as elegant or worth shipping.

#### 4.2.4 Domain and Expertise

Areas of deep knowledge, specialized vocabulary, expected depth and technical complexity of responses.

Example signals: technical vocabularies fluent in, citation patterns, depth of follow up questions in a given domain, comfort with jargon, meta knowledge about own expertise gaps.

#### 4.2.5 Temporal and Context

How needs and thinking shift based on project phase, urgency, and working mode.

Example signals: short term tactical vs long term strategic balance, recurring constraints by phase, deadline pressure response, context switching rate, active project context.

#### 4.2.6 Collaboration and Relationship

How the subject communicates with different audiences, formality shifts, and relationship context.

Example signals: directness gradient by audience (reports vs peers vs leadership vs vendors), information density, escalation patterns, trust calibration patterns.

#### 4.2.7 Decision and Risk

How the subject commits to positions and actions. Evidence thresholds, risk tolerance per domain, reversibility preferences, response to options versus direct recommendations. The "how you decide" signals, distinct from the "how you work" signals in Behavioral (Section 4.2.2).

Example signals: evidence thresholds before committing by domain, reversibility preferences, default escalation behavior, comfort with one way doors vs reversible moves, preference for option sets vs single recommendations, posture under ambiguity (delay, ask, default).

The per-signal shape for this layer is **normatively specified** (resolving open question 13 for the Decision and Risk layer first — see `spec/schemas/decision-risk-signal.schema.json`): each signal carries `dimension`, `polarity`, `value`, `confidence`, and `evidence_basis` (`observed` | `corroborated` | `self_reported`). A consumer SHOULD weigh signals by `confidence` and trust `observed`/`corroborated` above `self_reported`. The OPTIONAL `domain` field scopes a signal to a domain (same substring namespace as `action_descriptor.domain`, Section 10.2): the reference producer emits it as a **prototype** — a session's Decision and Risk signals are tagged only when one domain clearly dominates the subject's own messages, ambiguous sessions stay untagged — and a consumer prefers a matching-domain signal, falls back to an untagged one, and MUST NOT borrow a signal tagged for a different domain. Untagged signals keep fingerprint-wide semantics, so existing fingerprints are unaffected. `threshold` remains reserved (not yet emitted). The other seven layers' signal shapes remain implementation-defined in v0.1.

#### 4.2.8 Big Picture

How the subject sees themselves as a thinker and collaborator. Surfaces what the structured layers miss.

Example signals: recurring strategic frames, stable principles, self description that holds across contexts, multi year goal coherence, narrative the subject uses to explain their own work.

### 4.3 Signal Polarity

Signals within any layer carry one of two polarities:

- **Positive (preference).** Patterns the subject exhibits or values that align with the subject's working style.
- **Negative (anti-preference).** Patterns the subject avoids, rejects, or stops on. Producers SHOULD record these explicitly rather than relying on the absence of a positive signal.

Anti-preferences are diagnostic. Most subjects can articulate what they reject in AI output faster and more confidently than what they prefer. The extraction algorithm and any user facing intake (interview, settings, onboarding) SHOULD prompt for both polarities. Negative space is as load bearing as positive space.

Signal polarity is recorded in the per-signal schema, which is **normative for all layers** as of v0.1.17 (`fingerprint.schema.json#/$defs/signal`, generalizing the §4.2.7 Decision & Risk specialization; open question 13 resolved). When constructing an injection projection, consumers MUST honor anti-preference signals. If a layer summary or positive signal in a layer contradicts an anti-preference signal in the same layer, the anti-preference takes precedence.

Anti-preferences MAY also be authored explicitly by the subject through the user configuration (Section 5). When the same anti-preference appears in both the fingerprint (inferred) and the user configuration (authored), they reinforce each other; consumers SHOULD treat the authored form as canonical for the purpose of resolving conflicts.

### 4.4 Persona Strength

`persona_strength` at the document level is a producer computed scalar. The protocol does not specify how it is computed (the algorithm is proprietary). Consumers MAY use it to gate features (for example, an injection adapter MAY refuse to inject a fingerprint with strength below a configured threshold).

### 4.5 Extensions

The `extensions` field is reserved for producer specific or experimental data that does not yet have a standard home. Producers SHOULD prefix extension keys with a producer identifier (for example, `phaedo_io.x_experimental_dream_journal`). Consumers MUST ignore unknown extension keys.

Extensions are not part of the canonical document for signing purposes. See Section 6.

### 4.6 Layer Admission

All eight layers (Section 4.2) are **behavioral / persona** layers: each admits structured signals describing *stable patterns* in how the subject communicates, works, decides, and thinks. They MUST NOT carry **episodic content** — facts about the subject's life, environment, or specific entities (names, URLs, email addresses, domains, account identifiers, dates of specific events). Episodic facts are a separate, opt-in **content scope** outside this specification; a fingerprint is a model derived from observation, not a memory of conversations (Section 2).

This boundary is the protocol's privacy and differentiation property: a fingerprint that leaks episodic facts into its behavioral layers has rebuilt content memory with extra steps. Producers MUST NOT emit a signal whose `value` is an episodic fact; consumers and validators SHOULD reject one. A **conformant fingerprint** carries no URL, email address, or specific domain in any signal value. The reference producer enforces this (it drops such values from the projection — `extraction/fingerprint-update.js`) and the reference validator flags them as errors (`spec/validate-fingerprint.mjs`, high-precision URL/email/domain markers).

The per-layer **signal field vocabulary** — which `field` values each layer canonically carries — is published as a normative reference in `spec/layer-vocabulary.json` (generated from, and kept in sync with, the reference producer's catalog). An unknown `field` is not itself a conformance failure (the vocabulary evolves), but episodic *content* in any value is.

### 4.7 Conflict Resolution

A layer's `signals[]` is an **evidence log**: it MAY hold several signals for the same `field` with different `value`s, learned from different sources at different times. The array is therefore the provenance record — retained in full. But a consumer needs **one resolved value per dimension** to act on, and a projection (Section 9) that asserts two conflicting values for one dimension with no resolution is a defect — most dangerously for decision-relevant dimensions an agent consults before acting (e.g. autonomy). This section defines the **deterministic** resolution so every conformant implementation computes the same resolved value from the same evidence log. (User-*authored* configuration — Section 5 / Section 5.2 `standing_rules` / Section 10.5 — overrides any inferred resolution; this section governs inferred signals only. A `standing_rules` instruction carrying a `decision` binding for a dimension makes that dimension authored-canonical for the projection, ahead of any inferred resolution computed here.)

**Polarity precedence (Section 4.3) is applied first**: within a field, a negative (anti-preference) signal is the effective one over a positive. Value resolution then runs over the effective set.

**Evidence-tier precedence with a minimum-evidence floor.** Rank candidate signals by `evidence_basis`: `corroborated` > `observed` > `self_reported` (an absent/unknown basis ranks lowest). Within a tier, higher `observation_count` wins, then more recent `last_observed`, then a stable order on the value (full determinism — no ties). The **resolved value** is the top-ranked candidate, with one safeguard:

> **Minimum-evidence floor.** An `observed`/`corroborated` signal **overrides** a contradicting `self_reported` (interview) value only once it has accumulated `observation_count ≥ RESOLUTION_FLOOR` (normative default **3**). Below the floor, a genuine conflict between inferred and self-reported values resolves to **`status: "unresolved"`** rather than confidently flipping. A consumer MUST treat an `unresolved` dimension conservatively — for a consultation (Section 10) this means biasing toward `clarify`/`escalate`, never reading it as a confident instruction.

The floor exists because the most dangerous failure is *thin* observed evidence overriding a stated preference: two stray observations should not flip "check before acting" into "execute without dialogue" for an autonomous agent. Agreement across sources (same value) is never a conflict and always resolves.

**Output.** Resolution yields, per dimension: the resolved `value`, its `evidence_basis` and `confidence`, a `status` of `resolved` or `unresolved`, and a `provenance` array of all candidate signals. Provenance requires no new wire structure — it *is* the `signals[]` log. The reference algorithm is `resolveDimension()` / `resolveLayer()` in `spec/validate-fingerprint.mjs`; test vectors are `spec/test-vectors/resolution.json` (category TC-RESOLVE). See Proposal 0006.

> **Conflict records (optional staging surface).** A producer MAY make an `unresolved` contradiction *durable and reviewable* by staging it in an optional `phaedo_fingerprint.conflict_records[]` queue when it detects, at reconcile time, a new `observed` value contradicting a `self_reported` one below the floor. A record is `{ conflict_id, layer, field, domain?, status: "open" | "resolved" | "dismissed", detected_at, last_seen, resolved_at?, candidates[] }`, where `candidates[]` is the both-sides provenance. Like `suggested_rules[]` (§5.2 note), this is a producer-written **proposal queue**: never injected, never consulted, derived from the evidence log (no wire change; `phaedo_protocol_version` stays `0.1`). It auto-closes (`resolved`) once the inferred side crosses the floor; a subject's `dismissed` is sticky. It exists so an autonomous agent's most decision-relevant contradictions are auditable and reviewable rather than silently collapsed by an upstream confidence merge. **Injection suppression (normative for the reference producer):** while a conflict is `open`, the producer **withholds that dimension from the projection** (Section 9) — the reference producer's `renderLayerSummary` omits an open-conflict `(field[, domain])` — so a contested value is **not asserted** to a consuming agent until the subject settles it (settling authors a canonical rule; dismissing un-suppresses to the confidence-merged value). Reference: `detectConflicts()` / `reconcileFingerprint()` / `renderLayerSummary()` in `extraction/`, locked by `scripts/test_conflict_records.mjs`. **Consult suppression (the symmetric half, §10):** the reference resolver likewise **withholds an open-conflict dimension from the consultation cue readers** — a contested cue is treated as no coverage, so a consult whose every relevant dimension is contested returns `insufficient_signal` (abstain → escalate to the subject) rather than a confident answer on an unsettled dimension. The model-judge path inherits this by reasoning over the already-suppressed projection. Reference: `mcp/consult-core.js` (`openConflictKeys` / `isFieldContested`), locked by `mcp/test-consult.mjs`.

## 5. User Configuration

The user configuration is a sibling artifact to the fingerprint. It carries information the subject authors directly rather than information inferred from observation.

The fingerprint is a behavioral model. The user configuration is authored content. Both belong to the same subject, both are encrypted under the same key material, both ride the same envelope, and both are injected at session start. The protocol treats them as separate artifacts so that authored content is not reverse engineered from behavior, and so that inferred patterns do not have to share a structure with explicit rules.

Authored content has different properties than inferred content:

- It does not get learned over time. It is stated once and edited when needed.
- It does not have a producer assigned confidence score.
- It does not improve with more observations.
- It is applied verbatim, not summarized.

Treating it as a layer of the fingerprint creates category confusion. Treating it as a sibling keeps each artifact's semantics clean.

### 5.1 Top Level Document

```json
{
  "phaedo_protocol_version": "0.1",
  "configuration_id": "c2a8d4f1-9e6b-4c2a-8d4f-1e6b4c2a8d4f",
  "subject_id": "opaque_subject_identifier",
  "fingerprint_id": "f1b9c4e2-7d3a-4e8b-9c0f-2a1b3c4d5e6f",
  "created_at": "2026-04-28T17:00:00Z",
  "updated_at": "2026-04-28T17:00:00Z",
  "schema_revision": 1,
  "standing_instructions": [],
  "example_corpus": [],
  "extensions": {}
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `phaedo_protocol_version` | string | yes | Protocol version this document conforms to. |
| `configuration_id` | string (UUID v4) | yes | Stable identifier for this configuration instance. |
| `subject_id` | string | yes | Same opaque identifier used by the paired fingerprint. |
| `fingerprint_id` | string (UUID v4) | yes | The fingerprint this configuration is paired with. |
| `created_at` | string (RFC 3339) | yes | UTC timestamp of configuration creation. |
| `updated_at` | string (RFC 3339) | yes | UTC timestamp of most recent update. |
| `schema_revision` | integer | yes | Integer revision counter. |
| `standing_instructions` | array | yes | Ordered list of standing instructions. See Section 5.2. |
| `example_corpus` | array | yes | Reference outputs marked exemplary. See Section 5.3. |
| `extensions` | object | no | Reserved for producer specific extensions. Same rules as fingerprint extensions (Section 4.5). |

### 5.2 Standing Instructions

A standing instruction is an explicit rule the subject would state to every AI session if they remembered. Standing instructions are the highest leverage user configuration content because they are direct, authored, and applied verbatim.

> **v0.1 A-home — `standing_rules` (Proposal 0007).** The §5 User Configuration artifact is the eventual home, but it is not yet built. For v0.1, standing instructions are carried on the fingerprint as `phaedo_fingerprint.standing_rules` (the same pragmatic A-home §10.5 policies use), so they travel and inject today with no new artifact; they migrate to this sibling in v0.2. A `standing_rules` entry is this §5.2 object plus a `kind` discriminator (`instruction` | `authorization`, per Proposal 0003) and an optional `decision` binding to a Decision & Risk dimension (`{dimension, polarity, domain}`); a bound rule is authored-canonical for that dimension (§4.7). The shape is `fingerprint.schema.json#/properties/phaedo_fingerprint/properties/standing_rules`. Authored-only is structural: the reference extraction pipeline has no write-path into `standing_rules` (locked by `scripts/test_standing_rules_invariant.mjs`).

> **Delta promotion (the steady-state seed).** Onboarding (the `rule_elicitation` flow) seeds the layer at cold start; after that, the producer MAY *propose* rules without ever authoring them. When an inferred Decision & Risk signal becomes established (observed/corroborated, `observation_count ≥ 3`, no rule yet on that dimension), it is staged as a `suggested` candidate in `phaedo_fingerprint.suggested_rules[]` — a proposal queue, **not** authored content, never injected or consulted. The subject reviews and **endorses** (writing a `standing_rules` entry with `source:"delta_promotion"`) or rejects (kept `dismissed`, not re-proposed). Endorsement is the only write into the authored array, so authored-only holds through the loop. Reference: `extraction/rule-promotion.js` (stage) + `rule-review.js` (endorse/reject).

Examples (illustrative, not normative):

- "Use prose by default. Do not use bullet lists unless I ask."
- "Always show me failure modes alongside happy paths."
- "Default to skeptical pushback on my own ideas before agreeing."
- "Refer to me as Randy."
- "Do not use the word leverage as a verb."

Each instruction is an object:

```json
{
  "instruction_id": "i1a2b3c4-d5e6-7f89-0a1b-2c3d4e5f6789",
  "text": "Use prose by default. Do not use bullet lists unless I ask.",
  "scope": "global",
  "priority": 100,
  "created_at": "2026-04-28T17:00:00Z",
  "updated_at": "2026-04-28T17:00:00Z"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `instruction_id` | string (UUID v4) | yes | Stable identifier. |
| `text` | string | yes | The rule statement. UTF-8, NFC normalized. v0.1 limit: 1000 characters. |
| `scope` | string | yes | One of `global`, `domain`, `task`. v0.1: only `global` is normative. `domain` and `task` are reserved for v0.2. |
| `priority` | integer | yes | Ordering hint. Lower values inject first. Ties broken by `created_at`. |
| `created_at` | string (RFC 3339) | yes | UTC timestamp. |
| `updated_at` | string (RFC 3339) | yes | UTC timestamp. |

Standing instructions are injected verbatim. Producers MUST NOT summarize, paraphrase, or interpret them. Consumers MAY omit instructions that exceed the session's token budget but MUST omit them in priority order, dropping highest priority value (lowest precedence) first.

### 5.3 Example Corpus

The example corpus is a small set of representative outputs the subject has marked as exemplary. The corpus answers "what does good look like for me" through samples rather than rules.

Each example is an object:

```json
{
  "example_id": "e2b3c4d5-e6f7-89a0-b1c2-d3e4f5a6b7c8",
  "kind": "writing_sample",
  "label": "Investor update, calm tone",
  "content": "<the sample content>",
  "tags": ["voice", "investor"],
  "created_at": "2026-04-28T17:00:00Z"
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `example_id` | string (UUID v4) | yes | Stable identifier. |
| `kind` | string | yes | One of `writing_sample`, `decision_record`, `code_sample`, `other`. |
| `label` | string | yes | Short description for indexing and display. |
| `content` | string | yes | The sample. UTF-8. v0.1 limit: 8000 characters per example. |
| `tags` | array of strings | no | Producer or user assigned tags for retrieval. |
| `created_at` | string (RFC 3339) | yes | UTC timestamp. |

v0.1 specifies a soft cap of 20 examples per corpus and 8000 characters per example. Consumers SHOULD treat the corpus as a retrieval source rather than a verbatim dump: select the most relevant examples for a given session against session context (Section 9.2) rather than injecting all of them.

### 5.4 Pairing with the Fingerprint

A user configuration is paired with exactly one fingerprint via `fingerprint_id`. A fingerprint MAY exist without a paired configuration (a subject who has not authored any standing instructions or examples). A configuration without a paired fingerprint is invalid in v0.1.

When both are present, consumers SHOULD inject both at session start. The injection adapter assembles a combined projection containing:

1. Standing instructions, in priority order.
2. Selected layer projections from the fingerprint (Section 9.4).
3. Selected examples from the corpus, retrieved against session context.

Order matters. Standing instructions appear before fingerprint material in the assembled projection. They are explicit rules and SHOULD take precedence when authored configuration conflicts with inferred patterns.

### 5.5 Encryption and Integrity

The user configuration uses the same envelope (Section 7), canonical serialization (Section 6), and integrity scheme (Section 8) as the fingerprint. The same master key material MAY be used, with a distinct domain separation context (`phaedo:config:enc:v0.1`) to derive the configuration encryption key. This keeps fingerprint and configuration ciphertexts independent under the same root key, so a leak of one does not compromise the other.

### 5.6 Authoring Surface

The protocol does not specify a user authoring surface. Producers MAY expose any interface (text editor, structured form, voice capture, prompt rewriter). v0.1 places one constraint on producers: standing instructions and examples MUST be confirmed by the subject before being committed to the configuration. Inferred edits are forbidden. Configuration is authored content by definition.

## 6. Canonical Serialization

For signing and integrity checks, an artifact document (fingerprint or user configuration) MUST be serialized in a canonical form so that producer and consumer compute identical bytes from identical content. The canonicalization rules in this section apply to every artifact kind.

> **Implementation status (v0.1, 2026-06-15).** RFC 8785 (below) is the **target** canonicalization for the **signing path**, and §8.3 asymmetric signing is **deferred to v1.0** — so no v0.1 artifact body is yet signed over canonical bytes (the encrypted artifact body is serialized with plain `JSON.stringify`; confidentiality + integrity come from the AES-GCM envelope, §7). The **one** canonicalization v0.1 actually performs is the **envelope metadata AAD binding** (§7.3), via the reference `Phaedo.Envelope.canonicalStringify` — a **JCS-compatible subset**: object-key ordering by UTF-16 code units, recursive application, array-order preservation, UTF-8, and no insignificant whitespace all match RFC 8785. **Not yet implemented:** RFC 8785 number canonicalization for exponential / edge-magnitude values (the subset defers number formatting to `JSON.stringify`) and RFC 8785 minimal control-character escaping. These are byte-equal to RFC 8785 for the value types v0.1 metadata and artifacts actually contain (strings, simple decimals/integers, booleans, null, arrays, nested objects), and the full method lands with §8.3 signing. The subset's behavior is **locked by conformance vectors** — `spec/test-vectors/canonicalization.json` (category TC-CANON), run by `spec/test-vectors.mjs`.

### 6.1 Canonicalization Method

v0.1 specifies **JSON Canonicalization Scheme (RFC 8785)** as the canonicalization method.

Implementations MUST:

1. Build a JSON object containing all required top level fields of the artifact being serialized. For a fingerprint, this includes any present layer documents (Section 4.2). For a user configuration, this includes the standing instructions array and the example corpus array (Section 5).
2. Exclude the `extensions` field from the canonical form. Extensions are explicitly out of scope for signing in v0.1.
3. Apply RFC 8785 canonicalization to the resulting object.
4. Use the resulting byte sequence (UTF-8) as input to encryption and integrity operations.

### 6.2 String Encoding

All strings are UTF-8. Implementations MUST normalize text to NFC before canonicalization to avoid signature breakage from equivalent but byte different strings.

### 6.3 Numbers

Numeric fields are serialized per RFC 8785 number rules. Real number fields in `[0.0, 1.0]` (fingerprint `persona_strength`, per layer `confidence`) SHOULD be expressed with three decimal places of precision in canonical form. Integer fields (such as `schema_revision` and standing instruction `priority`) are serialized as exact integers.

## 7. Encryption Envelope

### 7.1 Algorithm

v0.1 specifies **AES-256-GCM** as the only normative encryption algorithm. Implementations MUST support it. Future versions may add algorithms; the algorithm identifier field exists to support that.

### 7.2 Envelope Structure

The envelope is a JSON object:

```json
{
  "phaedo_envelope_version": "0.1",
  "alg": "AES-256-GCM",
  "iv": "base64url(12 bytes)",
  "kdf": {
    "alg": "PBKDF2-HMAC-SHA-256",
    "salt": "base64url(16 bytes)",
    "iterations": 600000
  },
  "ciphertext": "base64url(...)",
  "tag": "base64url(16 bytes)",
  "metadata": {
    "artifact_kind": "fingerprint",
    "fingerprint_id": "f1b9c4e2-7d3a-4e8b-9c0f-2a1b3c4d5e6f",
    "created_at": "2026-04-28T17:00:00Z"
  }
}
```

For a user configuration envelope, `artifact_kind` is `user_configuration` and the metadata carries `configuration_id` and `fingerprint_id` (the paired fingerprint) instead of `fingerprint_id` alone:

```json
{
  "phaedo_envelope_version": "0.1",
  "alg": "AES-256-GCM",
  "iv": "...",
  "kdf": { "...": "..." },
  "ciphertext": "...",
  "tag": "...",
  "metadata": {
    "artifact_kind": "user_configuration",
    "configuration_id": "c2a8d4f1-9e6b-4c2a-8d4f-1e6b4c2a8d4f",
    "fingerprint_id": "f1b9c4e2-7d3a-4e8b-9c0f-2a1b3c4d5e6f",
    "created_at": "2026-04-28T17:00:00Z"
  }
}
```

Fields:

| Field | Required | Description |
|---|---|---|
| `phaedo_envelope_version` | yes | Envelope format version. |
| `alg` | yes | Algorithm identifier. v0.1 only `AES-256-GCM`. |
| `iv` | yes | 96 bit initialization vector, base64url encoded, no padding. MUST be unique per encryption with the same key. |
| `kdf` | conditional | Required when the encryption key is derived from a passphrase. Omitted when an externally managed key (such as a secure enclave wrapped key) is used. |
| `kdf.alg` | yes (when present) | KDF identifier. v0.1 only `PBKDF2-HMAC-SHA-256`. |
| `kdf.salt` | yes (when present) | 128 bit salt, base64url encoded. |
| `kdf.iterations` | yes (when present) | Iteration count. v0.1 minimum: 600,000. |
| `ciphertext` | yes | AES-256-GCM ciphertext over the canonical artifact bytes (Section 6). |
| `tag` | yes | 128 bit GCM authentication tag. |
| `metadata` | yes | Cleartext metadata for routing and indexing. Does NOT include any artifact content. |
| `metadata.artifact_kind` | yes | One of `fingerprint`, `user_configuration`. Identifies which artifact the envelope wraps without requiring decryption. |
| `metadata.fingerprint_id` | conditional | Required for `fingerprint` envelopes. Required for `user_configuration` envelopes (identifies the paired fingerprint). |
| `metadata.configuration_id` | conditional | Required for `user_configuration` envelopes. |
| `metadata.created_at` | yes | UTC timestamp of envelope creation. |

### 7.3 Authenticated Associated Data

The canonical serialization of the `metadata` field MUST be passed as additional authenticated data (AAD) to AES-GCM. This binds the cleartext metadata to the encrypted body so a tampered metadata field is detected on decrypt.

### 7.4 Key Material

v0.1 supports two key material sources for the **encryption envelope** (wire / portable export):

1. **Passphrase derived.** The user supplies a passphrase. The producer derives a 256 bit key with PBKDF2-HMAC-SHA-256. The KDF parameters are recorded in the envelope.
2. **Enclave wrapped.** The 256 bit key is generated and stored in a secure enclave or TEE on the vault device. The enclave returns a wrapped key for transport between vault and working device. v0.1 does not normalize the wrapping format; this is deferred to v0.2.

Separately, for the **working-device at-rest cache** (distinct from the envelope above):

3. **Working-device cache key (browser-held).** A paired client (e.g. the browser extension) MAY encrypt its at-rest fingerprint cache under a per-install, **non-extractable** AES-256-GCM key held by the client itself (reference implementation: a WebCrypto `CryptoKey` in IndexedDB, owned by the service worker). This is **defense-in-depth for the derived cache only** — it is NOT the master key (which never leaves the vault device), and because the client can decrypt without the vault device it provides neither phone-gated access nor forensic-grade protection. The cache is recoverable from the vault device on key loss (re-sync). See `docs/roadmap/extension-at-rest.md` for rationale and the (rejected) alternatives.

### 7.5 Lock Scope and Key Caching

v0.1 leaves lock scope and key caching policy to the implementation, with the following constraint: the working device MUST NOT retain plaintext key material in process memory for longer than the duration of an active injection or consultation. After the operation completes, the key MUST be zeroed or released.

> **Open decision.** Phase 0 has four open encryption decisions (lock scope, key caching, migration flow, locked fallback). These will be specified normatively in v0.2 once the prototype work answers them.

## 8. Integrity and Portable Export

### 8.1 In Place Integrity

AES-GCM provides authenticated encryption. Decrypt failure MUST be treated as an integrity failure and MUST NOT be retried with degraded checks.

### 8.2 Portable Export

An artifact exported for portability — for example, encoded in a QR code for device to device handoff — MUST be wrapped in a portable export object. Both fingerprint and user configuration use the same wrapper. The `mode` field discriminates two export modes with different trust models. When exporting a paired set, producers MAY emit two export objects (one per artifact) or a single combined export object that contains both; v0.1 specifies only the single artifact form below. The combined form is deferred to v0.2.

**Bootstrap mode.** Bootstrap export seeds a vault device that has no prior pairing with the producer — the first handoff, before any shared key exists. Because no key material has been established, the artifact is carried in cleartext:

```json
{
  "phaedo_export_version": "0.1",
  "mode": "bootstrap",
  "artifact": { ... },
  "pairing_token": "<opaque producer-generated token>",
  "digest": "base64url(32 bytes)"
}
```

`artifact` is the cleartext artifact document (Section 4 or Section 5). `digest` is SHA-256 over the canonical serialization (Section 6) of the export object with the `digest` field omitted. The digest is keyless: it gives tamper-evidence and detects transport corruption — for example, mis-assembled QR chunks — but NOT authenticity. It does not prove who produced the export.

`pairing_token` is an opaque value the producer also retains locally. The scanning device echoes it in the subsequent pairing handshake, where Section 7 key material is established, binding the out-of-band scan to the authenticated channel. Once that handshake completes, all further transfers use backup mode or the authenticated session channel — the cleartext window is exactly one scan wide.

A producer MAY include additional fields carrying its own pairing-handshake material — for example, `ext_pub`, an identity public key the consumer uses for out-of-band MITM detection. The digest covers every field except `digest` itself, so any such field is integrity-checked on receipt.

Bootstrap mode's trust model is physical: the export is displayed transiently, in the subject's presence, and scanned by a device the subject controls. The security boundary is camera line of sight. A producer MUST NOT use bootstrap mode for at-rest storage or for any transfer that is not a live, in-person handoff; a persisted bootstrap object is an unprotected plaintext artifact.

**Backup mode.** Backup export is used once a vault device and its master key exist — re-exporting an artifact to another device or to durable storage. The artifact is wrapped in a Section 7 envelope:

```json
{
  "phaedo_export_version": "0.1",
  "mode": "backup",
  "envelope": { ... },
  "hmac": "base64url(32 bytes)"
}
```

The `hmac` field is HMAC-SHA-256 over the canonical serialization of the `envelope` object, computed with a key derived from the master key via the distinct domain separated KDF context `phaedo:export:hmac:v0.1`. The HMAC is informative, not a substitute for the envelope's GCM tag: its purpose is to detect transport corruption and confirm that the export was produced by a holder of the master key, before decryption is attempted.

**Mode selection.** A producer MUST use backup mode whenever master key material is available to it. Bootstrap mode is permitted only for the initial pre-pairing handoff.

Bootstrap mode's cleartext trust model is camera line of sight (above). A producer MUST NOT use bootstrap mode for a handoff that does not transit a private out-of-band channel the subject controls — in particular, it MUST NOT carry the cleartext artifact over a relay or any other infrastructure path. The pairing-code rendezvous in Section 8.4 is the relay-safe alternative.

### 8.3 Asymmetric Signing (Deferred)

v0.1 does not specify asymmetric signing. v1.0 will introduce Ed25519 signatures for fingerprints intended to be presented to third parties (for example, an enterprise context where an employer wants to verify that an employee's fingerprint was produced by a specific Phaedo instance). The schema reserves a `signatures` field for this; v0.1 producers MUST NOT populate it.

### 8.4 Pairing-Code Rendezvous and Encrypted Seed

A producer MAY establish the initial pairing without a QR handoff. In this mode the out-of-band channel is a short, single-use **pairing code** the subject reads from the producer and types into the vault device, rather than a scanned code.

The pairing code is a rendezvous and correlation value only. It carries no key material and no artifact. Its sole functions are (1) to let the two devices find each other (for example, as the lookup key for a relay offer) and (2) to bind the subsequent pairing handshake to this attempt. It is not a confidentiality control and not the man-in-the-middle control; the Short Authentication String (SAS) derived in the Section 7 handshake is. A producer MUST generate the code with a cryptographic RNG, give it a short expiry, and accept it for exactly one pairing.

Two rendezvous channels are defined, distinguished by whether the pairing handshake reaches the vault device directly or through untrusted infrastructure:

- **Direct (local network).** When the producer reaches the vault device over a local channel it can enumerate itself (for example, LAN discovery of the vault's pairing endpoint), the vault device MAY accept the pairing handshake without a pre-shared code: physical local-network access plus the user's SAS confirmation is the gate. This is the no-typing path.
- **Relayed.** When the handshake is carried by a relay or any channel reachable beyond the local network, the vault device MUST require a matching pairing code before presenting an SAS confirmation, so that an arbitrary remote party cannot trigger confirmations. This is the typed-code fallback.

In both channels the SAS is the man-in-the-middle control and the user's confirmation is the authorization. A vault device MUST NOT revoke or disturb an existing pairing on the basis of an unconfirmed pairing request. A vault device MAY hold **multiple concurrent verified clients** (for example, a browser extension and a local MCP server); confirming a new pairing **adds** a client and MUST NOT revoke or displace any other. Each client is approved independently by its own SAS confirmation and is independently revocable; a client is removed only by explicit user revocation, or by denial of its own pairing request.

Because no QR carries the producer's identity public key, that key (`ext_pub`) travels in the pairing handshake request itself. Substitution of either party's long-term public key in transit is detected by the SAS: both devices derive the SAS from both long-term public keys and the subject compares the two displayed strings before confirming. A producer and consumer MUST surface the SAS for comparison and MUST NOT complete pairing until the subject confirms it.

The artifact is not carried during rendezvous. Once the handshake establishes an SAS-verified `pair_key`, the producer transfers the artifact as a **seed envelope**: a Section 7.2 encryption envelope delivered to the vault device over the authenticated channel (for example, a dedicated pairing endpoint). The envelope's metadata carries `artifact_kind` and, for a fingerprint, `fingerprint_id`, bound as AAD. The consumer MUST treat a decryption failure as an integrity failure (Section 8.1) and MUST reject a seed from a client that has not been confirmed (`verified`). Because the artifact is encrypted under a key the relay never holds, no cleartext artifact crosses relay infrastructure — the confidentiality property the camera channel previously provided.

The seed envelope's key is `seed_key`, a domain-separated sub-key of the pair key:

```
seed_key = HKDF-SHA-256(ikm = pair_key, salt = "", info = "phaedo-seed-v0.1", L = 32)
```

`seed_key` is derived from `pair_key` rather than reusing it directly so that the AES-256-GCM seed key is never the same octets as the HMAC key `pair_key` carries in the Section 7 authorize exchange. Both parties re-derive `seed_key` on demand from the stored `pair_key`; it is not transmitted.

```json
{
  "phaedo_envelope_version": "0.1",
  "alg": "AES-256-GCM",
  "iv": "base64url(12 bytes)",
  "ciphertext": "base64url(...)",
  "tag": "base64url(16 bytes)",
  "metadata": { "artifact_kind": "fingerprint", "fingerprint_id": "<uuid>", "created_at": "<rfc3339>" }
}
```

The seed envelope reuses the Section 7.2 structure verbatim; only the key selection (the `pair_key` rather than a session or master-derived key) and the one-time pre-seed use distinguish it.

## 9. Injection Protocol

The injection protocol describes how a consumer requests a fingerprint projection suitable for inclusion in an AI session as system prompt or context.

### 9.1 Injection Modes

Two modes are defined. Both modes operate on the combined projection (selected fingerprint layers, standing instructions from the paired user configuration, and selected examples; see Section 5.4).

1. **Standard mode.** The injection adapter receives plaintext content: layer projections, standing instructions verbatim, and selected examples. This is the default and is what the reference Chrome extension uses. As of Proposal 0007 the reference consumer (`context-block.js`) renders the standing instructions carried in `phaedo_fingerprint.standing_rules` (those with `text`) **verbatim** as a top-of-projection **"Operating rules (user-confirmed)"** block — above the inferred-style sections, so a confirmed rule wins where it conflicts with an inferred habit (§4.3), ordered by §5.2 `priority` and applied silently (not restated).
2. **Enhanced privacy mode (Stained Glass).** The injection adapter receives an obfuscated payload designed to preserve persona signal while removing identifying detail. The obfuscation algorithm is proprietary and out of scope for v0.1; the protocol specifies only the request flag and the response envelope. v0.1 does not specify whether or how standing instructions and examples are obfuscated under this mode; this is deferred to v0.2.

A consumer MAY request enhanced privacy mode. A producer MAY refuse if it does not support the mode. Refusal is a structured error, not a silent fallback to standard mode.

> **Note on configuration injection in the request and response shapes.** Section 9.2 and 9.3 show fingerprint-only fields. Substantive integration of configuration into the request and response is tracked as a separate decision (see change log v0.1.1) and is expected in v0.2.

### 9.2 Injection Request

```json
{
  "phaedo_request_version": "0.1",
  "request_type": "injection",
  "session_context": {
    "platform": "claude_web",
    "task_hint": "drafting a strategic memo",
    "audience_hint": "investor",
    "max_tokens": 2000
  },
  "mode": "standard",
  "requested_layers": ["all"],
  "consumer_id": "phaedo_chrome_extension_v0.1.3"
}
```

| Field | Required | Description |
|---|---|---|
| `request_type` | yes | One of `injection`, `consultation` (Section 10). |
| `session_context` | yes | Cleartext hints about the session. Used for layer selection. MUST NOT include subject content. |
| `mode` | yes | `standard` or `enhanced_privacy`. |
| `requested_layers` | yes | Array of layer names, or `["all"]`. |
| `max_tokens` | no | Soft cap on the size of the projection. Producer SHOULD honor; MAY exceed if a smaller projection would be unusable. |
| `consumer_id` | yes | Identifier of the requesting software. |

### 9.3 Injection Response

```json
{
  "phaedo_response_version": "0.1",
  "request_type": "injection",
  "fingerprint_id": "f1b9c4e2-7d3a-4e8b-9c0f-2a1b3c4d5e6f",
  "mode": "standard",
  "projection": "<plaintext system prompt content>",
  "layers_included": ["surface", "domain_and_expertise", "decision_and_risk", "big_picture"],
  "persona_strength": 0.62,
  "expires_at": "2026-04-28T18:00:00Z"
}
```

The `projection` field is the content the consumer inserts into the AI session. For standard mode, this is human readable text. For enhanced privacy mode, this is the obfuscated representation; format is producer specific in v0.1.

`expires_at` is a soft hint. Consumers SHOULD discard the projection after expiry and request a fresh one for a new session.

### 9.4 Layer Selection

In v0.1, layer selection is producer driven. The producer reads the session context hints and chooses which layers to project. A future version may expose a layer selection negotiation protocol; v0.1 does not.

Producers SHOULD document the selection logic so consumers can predict behavior. The reference implementation publishes its selection rules separately.

### 9.5 Errors

Errors are returned as:

```json
{
  "phaedo_response_version": "0.1",
  "request_type": "injection",
  "error": {
    "code": "mode_not_supported",
    "message": "Enhanced privacy mode is not available in this build."
  }
}
```

Error codes for v0.1:

- `mode_not_supported`
- `layer_not_available`
- `decryption_failed`
- `persona_strength_below_threshold`
- `request_malformed`
- `internal_error`

## 10. Agent Consultation Protocol

> **Status:** Implemented in the reference MCP server (`phaedo_consult`, all four types; Section 10.5 policies; the `insufficient_signal` abstain outcome as of v0.1.9). The request (10.2) and response (10.3) shapes are **stable** and may be built against; committed to v1.0. *How* a consumer derives the signal — a deterministic rule engine, a local model judge, or both — is implementation, not protocol. The reference implementation separates a **pure deterministic resolver** (`mcp/consult-core.js`, `resolveConsultationCore(vd, request)` — the publishable open-core surface) from the optional local model judge and transport (`mcp/consult.js`).

The consultation protocol lets an autonomous agent query the fingerprint at decision points without ingesting the full fingerprint.

### 10.1 Consultation Types

| Type | Purpose |
|---|---|
| `action_approval` | Should the agent proceed with a specific action? |
| `voice_draft` | Generate a draft in the subject's voice for a given task. |
| `domain_risk_check` | What is the subject's risk tolerance and evidence threshold in this domain? |
| `escalation_default` | When uncertain, what should the agent default to (proceed, ask, escalate)? |

### 10.2 Consultation Request

```json
{
  "phaedo_request_version": "0.1",
  "request_type": "consultation",
  "consultation_type": "action_approval",
  "agent_id": "vendor_agent_v2.4",
  "action_descriptor": {
    "domain": "financial",
    "reversible": false,
    "magnitude": "high",
    "summary": "Approve $12,000 vendor payment outside standard terms."
  },
  "context": {
    "deadline_pressure": "low",
    "evidence_provided": ["invoice", "po_match"]
  }
}
```

### 10.3 Consultation Response

```json
{
  "phaedo_response_version": "0.1",
  "request_type": "consultation",
  "consultation_type": "action_approval",
  "signal": "escalate",
  "confidence": 0.78,
  "rationale_hint": "Subject's pattern in irreversible financial actions above $5K is to require human review.",
  "deference_level": "low"
}
```

`signal` is one of:

- `proceed` — pattern aligns with action
- `proceed_with_note` — pattern aligns but flag for after the fact review
- `clarify` — agent should gather more evidence before acting
- `escalate` — surface to subject for live decision
- `decline` — pattern actively opposes action
- `insufficient_signal` — **abstain.** The producer declines to answer because the fingerprint lacks adequate coverage or confidence for this `action_descriptor` domain. Distinct from `clarify` (which asks the *agent* to gather more evidence and re-ask) and from an error (the request was well formed; the answer is "no basis"). A consumer SHOULD treat `insufficient_signal` as "no signal — fall back to your own default or escalate to the subject," not as a reason to retry. Conventionally paired with `deference_level: low`. A confidently wrong `proceed` is the worst output this protocol can produce; abstaining is preferred to guessing.

`deference_level` indicates how strongly the agent should weight the signal:

- `high` — strong pattern match, high confidence; agent SHOULD follow
- `medium` — pattern present but ambiguous; agent SHOULD weigh signal alongside other inputs
- `low` — sparse data; signal is a hint, not a directive

### 10.4 Consultation Boundaries

A consulting agent MUST NOT request the full fingerprint via the consultation API. The consultation API returns only signals, not layer content. Agents that need fingerprint content MUST go through the injection API and accept the corresponding boundaries.

### 10.5 Consultation Policies (deference tuning)

The subject MAY author standing **consultation policies** — per-domain / per-magnitude / per-reversibility guardrails that the consumer applies on top of the inferred signal. Policies are authored content (subject-confirmed, like §5 standing instructions), so they **override** the inferred consultation signal: they encode hard pre-decisions ("always escalate irreversible spend or legal", "auto-proceed low-stakes reversible writing") that make delegation safe. Policies travel with the subject (carried alongside the fingerprint) and are applied locally; they never expand the §10.4 boundary (the response shape is unchanged — policy provenance is conveyed in `rationale_hint`).

A policy document is a list of rules:

```json
{
  "version": "0.1",
  "rules": [
    { "id": "irreversible-spend-legal",
      "match": { "domains": ["financial", "legal", "payment"], "reversible": false },
      "effect": "escalate",
      "note": "Irreversible money or legal actions always go to a human" },
    { "id": "low-stakes-writing",
      "match": { "domains": ["writing", "draft", "email"], "magnitude_max": "medium", "reversible": true },
      "effect": "autoproceed",
      "note": "Low-stakes reversible writing is pre-approved" }
  ]
}
```

- **`match`** (all present constraints must hold): `domains` (substring match, case-insensitive, on the action descriptor's domain), `magnitude_min` / `magnitude_max` (`low` | `medium` | `high`), `reversible` (`true` | `false`), and `amount_gt` (a numeric threshold — matches only when the action descriptor carries an `amount` strictly greater than it, e.g. "escalate irreversible spend over 5000"; an action with no `amount` never satisfies a threshold rule).
- **`effect`**: `escalate` / `require_human` (force `escalate`), `decline` (force `decline`), `autoproceed` (force `proceed`); or `bias_cautious` / `bias_bold` (nudge the signal one step along `proceed → proceed_with_note → clarify → escalate`).
- **Conditional + revocable authority (optional).** `expires_at` (RFC 3339) makes an authorization **time-bound**: once it passes, the rule has LAPSED and no longer applies — an identity-free, fail-safe form of revocation (the inferred resolution simply stands). It is **enforced** by the reference resolver (a lapsed force or bias rule stops applying). `agents` (array of agent identities) scopes a rule to specific agents; it is **RESERVED, not yet enforced** — scoping on a self-asserted `agent_id` would be false security, so enforcement awaits an authenticated agent identity (a v0.2 concern alongside immediate revocation propagation to agents holding cached snapshots).
- **Force effects are authorizations.** A matched force effect is an authored, deterministic pre-decision: it resolves the consultation **canonically**, overriding the inferred signal, at `confidence: 1.0` and `deference_level: high`. Because it is authored content (subject-confirmed, §5.6 semantics), it is canonical over any inferred pattern, the same way an authored anti-preference is canonical over an inferred one (§4.3). A `bias_*` effect is a softer nudge and keeps the inferred confidence.
- **Precedence**: the first matching force-effect rule wins; otherwise the first matching bias applies. A force effect's `note` (or a default) is surfaced as the `rationale_hint`, so the agent sees *why* (e.g. "Per your standing rule: …").
- **Authored only.** Policies are authored content: a producer MUST confirm them with the subject before commit (§5.6) and MUST NOT infer or suggest them from observation. In the reference implementation the extraction pipeline has no write-path into `consult_policies` or `standing_rules` (locked by `scripts/test_consult_policies_invariant.mjs` and `scripts/test_standing_rules_invariant.mjs`).
- Policies MUST NOT apply to `voice_draft` (a redirect, not a decision).
- **Portability**: policies SHOULD be carried **inside the fingerprint** so they travel with it across every transport with no profile-schema change. As of Proposal 0007 the home is `phaedo_fingerprint.standing_rules` with `kind:"authorization"` (carrying the same `match`/`effect`); the standalone `phaedo_fingerprint.consult_policies` array is **DEPRECATED** but still honored as a back-compat read path (removed in v0.2 with the §5 migration). A consumer MAY also accept policies as a sibling artifact or local configuration.
- **Authored decision rules as a bias (Proposal 0007 §D).** A `standing_rules` entry of `kind:"instruction"` is a verbatim injected rule, not a gate; when it carries a `decision` binding with **negative** polarity, the reference resolver also projects it to a `bias_cautious` nudge for matching-domain (`decision.domain`) actions. **The asymmetry is by design and permanent: an authored disposition may make the agent MORE deferential, never less.** Autonomy *expansion* (auto-proceed) is never inferred from a soft rule — it requires an explicit `kind:"authorization"` with `effect:"autoproceed"` and a `match` (domain / magnitude / amount), so the relaxation is deterministic, scoped, and auditable. Positive-polarity and unbound instructions therefore inject only and have no consultation effect.
- **Deterministic pre-flight (binding affordance).** A consumer MAY expose a deterministic-only check of the standing authorizations — match the `action_descriptor` against the force rules, skipping all inferred evaluation — so an agent can short-circuit on a hard guardrail before a full consultation. The reference MCP binding exposes this as the `phaedo_check_authorization` tool; its response shape is `spec/schemas/authorization-check-response.schema.json` and it honors §10.4 (only the rule's decision and provenance cross). This is a transport convenience over the same authorizations, not a new resolution path.

### 10.6 Consultation Receipts

A consultation producer SHOULD keep an encrypted, **append-only**, vault-held record of every resolved consultation — the subject's audit trail of which agent asked what and what the oracle answered, and the data layer for agreement tracking. **Receipts never leave the device.** Phaedo (the company) and any intermediary hold nothing.

Each receipt captures the request descriptor and the response **signal fields only** — never layer content, never the projection, never rationale text (the §10.4 boundary holds inside the vault too):

```json
{
  "receipt_id": "uuid-v4",
  "created_at": "2026-06-10T17:00:00Z",
  "via": "phaedo_consult",
  "agent_id": "vendor_agent_v2.4",
  "request": {
    "consultation_type": "action_approval",
    "action_descriptor": { "domain": "financial", "reversible": false, "magnitude": "high", "amount": 12000, "summary": "Approve $12k vendor payment." }
  },
  "response": { "signal": "escalate", "confidence": 0.78, "deference_level": "high" },
  "user_action": null
}
```

Normative rules (machine form: `spec/schemas/consultation-receipt.schema.json`):

- **One receipt per resolved request**, abstains (`insufficient_signal`) included. Errors are not resolutions and produce no receipt. A matched deterministic pre-flight (§10.5 affordance) is a resolution and produces a receipt (`via` distinguishes it); an unmatched pre-flight is not.
- **`user_action` is the only mutable field.** `null` at write time; updated later when override data exists — `approved | rejected | modified | unknown` (what the subject actually did with the agent's action). Every other field is immutable after write. Receipts are never deleted individually; the store is bounded by a configurable cap (reference default 5,000) with oldest-first eviction, and is destroyed in whole with the account.
- **`drivers` (optional provenance, §10.4-safe).** An inferred consultation MAY record the Decision&Risk cues that drove it — `{cue, lean?/value?, confidence, basis}` — so the subject can later audit *why* the oracle answered as it did and calibrate it. This is a LOCAL field of the encrypted receipt only; it MUST NOT appear in the §10.3 response the agent sees (§10.4), and it carries direction/confidence/evidence-basis, never raw layer text. Absent for deterministic pre-flights and abstains (no inferred cue basis). Reference: `consultDrivers()` (`mcp/consult-core.js`).
- **Encryption:** the standard Section 7 AES-256-GCM envelope under a key derived with the **distinct domain-separation context `phaedo:receipts:enc:v0.1`** — the receipt store and any other at-rest artifact are independent ciphertexts under one root.
- **Purity of the resolver:** receipt writing happens in the transport layer after resolution, never inside the resolver. A receipt-write failure MUST NOT fail the consultation itself.
- **Signed receipts (future).** Post-v1.0, the reserved §8.3 `signatures` field is the intended path to receipts a third party can verify ("this agent did consult the subject's oracle and received this signal") without seeing the fingerprint. v0.1 producers MUST NOT populate it.

## 11. Conformance

### 11.1 Profiles and Levels

Profiles are grouped into three **conformance levels** by how much of the protocol an implementation must adopt. The level is the adoption floor: a higher level subsumes the capabilities of the lower ones.

| Level | Profile | Description | Required Sections | Crypto? |
|---|---|---|---|---|
| **0** | **Projection Consumer** | Reads an already-decrypted **projection** — the Section 9.3 response `projection` string, or the reference server's `phaedo://fingerprint/projection` resource — and inserts it into a session. Consumes a string. | 9.3 (response shape) | **None** |
| **1** | **Consumer** | Reads and decrypts fingerprints / configurations (passive). | 4, 5, 6, 7, 8 | Required |
| **1** | **Injector** | Decrypts artifacts and inserts projections into AI sessions. | 4, 5, 6, 7, 8, 9 | Required |
| **1** | **Agent Consultant** | Issues consultation queries and applies the constrained signal. | 4, 6, 7, 8, 10 (incl. 10.5) | Required |
| **2** | **Fingerprint Producer** | Creates and updates fingerprints. | 4, 6, 7, 8 | Required |
| **2** | **Configuration Producer** | Creates and updates user configurations. | 5, 6, 7, 8 | Required |
| **—** | **Full Reference** | All of the above. | All sections | Required |

**Level 0 is the floor — and it requires no Phaedo cryptography, canonicalization, or schema machinery.** A client that reads `phaedo://fingerprint/projection` (the reference MCP server's plaintext resource) or accepts the `projection` field of a Section 9.3 response and inserts it gets Phaedo compatibility by consuming a string. The no-knowledge encryption envelope (§7) is a property of the *producer/vault* boundary, not a prerequisite for *consuming* a projection a user has chosen to surface. This is the "conformance in an afternoon" tier and the intended on-ramp for the broadest set of implementers (any MCP host already qualifies by reading the resource).

For Level 1 and above, conforming fingerprint, consultation request/response, and policy documents validate against the machine-readable JSON Schemas in `spec/schemas/` (the normative machine form). The reference conformance check is `spec/test-schemas.mjs`; fingerprint documents additionally validate with `spec/validate-fingerprint.mjs`.

### 11.2 Mandatory and Optional Features

Mandatory for **Projection Consumer (Level 0)** — and *only* these:

- Accept the Section 9.3 response `projection` string (standard mode), or read the `phaedo://fingerprint/projection` resource, and insert it into the session.
- Tolerate unknown fields in the response / resource payload (forward compatibility).
- Cryptography (Section 7), canonical serialization (Section 6), and schema validation are **NOT required** at Level 0 — the projection is consumed as text. A Level 0 consumer never holds an encrypted artifact, so it has no envelope to open.

Mandatory for **all profiles at Level 1 and above**:

- AES-256-GCM encryption envelope (Section 7)
- RFC 8785 canonical serialization for signing inputs (Section 6) — see the §6 implementation-status note for the v0.1 subset
- All required top level document fields (Section 4.1)
- Tolerance of unknown extension keys (Section 4.5)

Mandatory for Injector:

- All five v0.1 error codes (Section 9.5)
- Standard mode injection (Section 9.1)

Optional for v0.1:

- Enhanced privacy mode (Section 9.1)
- Asymmetric signatures (Section 8.3, deferred)
- Specific KDF tuning above the v0.1 minimum (Section 7.2)

### 11.3 Self Identification

A conforming implementation MUST publish a conformance statement listing:

- Profile(s) and level(s) claimed
- Protocol version supported
- Optional features supported
- Test vector results (Section 12)

The reference implementation publishes its statement at **`spec/conformance.json`** (machine-readable). It is kept honest by `spec/test-schemas.mjs`, which ties it to reality: the declared `protocol_version` must match this document's version, and the test-vector categories it reports as passing must match the populated vector sets in `spec/test-vectors/`. A third party publishes its own `conformance.json` in the same shape.

## 13. Open Questions for v0.2

Tracked here so they are visible and resolved before v0.2:

1. **Enclave wrapped key format.** v0.1 leaves the wrapping format unspecified. v0.2 needs a normative format compatible with iOS Secure Enclave and Android StrongBox.
2. **Lock scope and key caching policy.** Phase 0 prototype work to drive a normative answer.
3. **Migration flow.** How a fingerprint moves between vault devices when the user replaces their phone.
4. **Locked fallback.** What an injection adapter does when the vault device is unavailable. Default behavior, user override, expiry of cached projections.
5. **User configuration scope expansion.** v0.1 standardizes only `global` scope for standing instructions. v0.2 should specify how `domain` and `task` scoped instructions activate based on session context.
6. **Example corpus retrieval.** v0.1 says consumers SHOULD treat the corpus as a retrieval source. v0.2 may specify a normative retrieval API or reranking signal for examples.
7. **Configuration vs fingerprint conflict resolution.** v0.1 says authored configuration takes precedence. v0.2 should specify exact resolution rules for finer grained conflicts (e.g., a standing instruction that contradicts a high confidence fingerprint signal in only one domain).
8. **Cross encoder reranking signal.** Phase 1 work introduces reranking at injection time. The protocol may need a hook for the consumer to pass a rerank request alongside an injection request.
9. **Asymmetric signature scheme.** Ed25519 confirmed for v1.0. v0.2 sketches the signing flow.
10. **Layer selection negotiation.** Does the consumer get a say, or is selection always producer driven? v0.1 chose producer driven; v0.2 may revisit.
11. **Consultation protocol formalization.** Move from sketch to normative for v1.0. *Progress:* `0001` abstain signal **folded v0.1.9**; `0003` authorizations (A-half) **folded v0.1.10**; `0005` `vd` contract **folded v0.1.12**; `0004` receipts **folded v0.1.13** (§10.6). Still queued: `0002` Decision & Risk per-signal schema (needs `extraction/` to emit the shape); `0003` B-half (§5 home for authorizations, v0.2).
12. **Stained Glass interface stability.** What hooks and metadata does the protocol need to expose so that third parties can implement compatible obfuscation modes without access to the proprietary algorithm?
13. **Per layer signal schemas.** ✅ *Resolved (v0.1.17).* The per-signal **shape** is now normative for **all eight layers** — `fingerprint.schema.json#/$defs/signal` (typed, bounded, enum `polarity`/`evidence_basis`), generalizing the Decision and Risk specialization in `decision-risk-signal.schema.json` (§4.2.7, `domain`/`threshold` reserved). The per-layer **field vocabulary** is published as a normative reference in `spec/layer-vocabulary.json` (generated from the reference producer catalog, tied by `spec/test-schemas.mjs`). Layer admission (§4.6) is now normative and enforced (no episodic content in behavioral layers). An unknown `field` remains tolerated (the vocabulary evolves); per-field `value` enums per layer are the remaining v0.2 tightening.
14. **Transport binding conformance.** `mcp-binding.md` is a draft transport binding. v0.2 should fold transport bindings into the conformance profiles (Section 11) and decide whether the MCP binding is normative for v1.0.
15. **Drift / change-point detection.** A behavioral model must distinguish a durable shift in a subject's pattern from a passing mood, and retire stale patterns (recency decay, change-point detection). v0.1 keeps `last_observed` for provenance only and applies no decay. Open: whether a normative recency/decay or change-point signal belongs in the resolver, or stays a producer-internal concern. Bites at scale; research track.
16. **Learning-loop integrity.** Corrections that update the fingerprint (delta annotations, agent-correction pairs) are not authenticated as genuinely the subject's, rate-limited only at the extraction-budget layer, and not anomaly-checked. An autonomous-agent future makes the *learning* surface (not just storage) a target. Open: what authentication / rate-limit / anomaly guarantees the protocol should require before a correction updates anything. Research track.
17. **Budget / blast-radius authorizations.** v0.1 §10.5 gates a single action by `amount_gt`. Long-running autonomous agents (Devin-style multi-day runs, agentic browsers, scheduled assistants) need standing **budgets** — `{amount, currency, window_hours, escalate_at}` carried on an authorization, decremented locally per resolved action, with `escalate_at` flipping the effect to `bias_cautious` and exhaustion flipping to `escalate`. Open: whether the running counter is a normative producer-internal artifact or a §10.6 receipt aggregate; window-reset semantics (rolling vs. fixed); how a budget composes with `agents` scope (per-agent budget envelopes). Roadmap target: v0.3. See `docs/roadmap/future-direction-2026-06-25.md`.
18. **Sub-agent capability tokens (scope snapshots).** When a parent agent spawns N children, N independent `phaedo_consult` calls are N× cost/latency AND a privacy expansion (every child re-pairs / re-derives). The future surface is a sealed, signed, time-bounded **scope snapshot** — "for the next 10 minutes, on these domains, magnitude ≤ medium, the answers are pre-baked as follows" — that a parent fetches once and hands to children. Composes with OQ9 (Ed25519): the parent's signature authorizes the scope; the child verifies. Open: scope grammar, revocation propagation when the underlying policy changes mid-snapshot, replay defense. Roadmap target: v1.0.
19. **Cross-agent delegated capabilities.** When agent A (vendor 1) hands off to agent B (vendor 2), B must reach the same fingerprint with bounded scope WITHOUT the subject re-pairing B from the vault. A **delegated capability** — user-signed, carrying `(delegating_agent, delegate_agent, domains, ttl)` — lets A authorize B for a slice of the consultation surface. Composes with OQ18 and with the §10.5 `agents` scope (RESERVED, blocked on authenticated agent identity). Open: revocation, transitive delegation (A→B→C: yes/no/depth-limited?), the audit chain on the §10.6 receipt. Roadmap target: v1.0.
20. **Voice / realtime capture binding.** Current capture covers chat surfaces only (the reference Chrome extension). The voice-agent class arriving 2026–27 (OpenAI Realtime, Vapi, Retell, Sesame, ElevenLabs Conversational) needs a binding spec: turn-taking signals, prosody/pace cues, interruption patterns → Surface / Behavioral / Collaboration & Relationship layer updates, honoring §10.4. Streaming-safe (no buffering of full conversation transcripts; deltas only). Open: the per-modality field vocabulary in `spec/layer-vocabulary.json`. Roadmap target: v1.0.
21. **On-device judge interface.** The reference resolver already supports an optional model-judge second pass beyond the deterministic resolver (`mcp/consult.js`). Specifying its interface as a normative *seam* lets third-party local models plug in — Apple Intelligence's on-device model, Qualcomm AI Hub, local Llama-class. Pure additive privacy gain: the judge stays local even when the calling agent is remote. Open: prompt shape, deterministic-fallback contract, judge-vs-resolver precedence (the resolver's `insufficient_signal` should not be silently overridden by a judge guess). Roadmap target: v1.0.
22. **Vault federation across devices (per-layer ACLs).** v0.1 assumes one vault per subject. A real user has a work device + a personal device; the protocol should accept **layered vaults** with per-layer authority objects ("work agents see `domain_and_expertise` + `temporal_and_context` only, never `big_picture`"). The Section 4.2 envelope already supports independent per-layer encryption — what is unspecified is the *authority* surface (who decides which layers a paired client sees). Open: canonical-vault resolution when layers disagree; whether the merge happens on the vault device or at consult time. Roadmap target: v1.x. Research-adjacent.
23. **Cross-fingerprint policy pre-flight (multi-party agent negotiation).** When two agents negotiate on behalf of two subjects (procurement: buyer's agent ↔ seller's agent; scheduling: two assistants; contract review: counter-party agents), each side needs to query the other's standing **authorizations** without learning persona content. `phaedo_check_authorization` extended to a counter-party endpoint, with a deterministic-only response and the §10.4 boundary preserved. The discovery layer (DID-style / DNS-style routing to a subject's oracle) is the missing piece, not the policy shape. Open: identity layer to bind to (W3C VC / DID, or a Phaedo-specific bootstrap). Roadmap target: v1.x. Speculative.

## Appendix B: Relationship to Other Standards

Informative.

- **OAuth 2.0 / OIDC.** Phaedo is to cognitive identity what OAuth is to authentication: a protocol that lets a user assert something to a third party in a verifiable, portable form, with the user controlling the canonical record.
- **W3C Verifiable Credentials.** Phaedo's signed fingerprint format (post v1.0) shares structural goals with VCs. v0.2 will evaluate whether the envelope can be expressed as a VC for interop.
- **JWE / JWS.** The envelope format is intentionally close to JWE in spirit but JSON object structured rather than compact serialization. v0.2 will evaluate alignment.
- **RFC 8785 (JCS).** Used directly for canonical serialization.
- **Model Context Protocol (MCP).** The primary transport binding for Phaedo's injection and consultation APIs, and the de facto integration standard for AI clients and agent frameworks. The Phaedo MCP server exposes the injection projection as an MCP resource and the consultation types as MCP tools. The binding is specified in `mcp-binding.md`; the core protocol remains transport-agnostic.

## Appendix C: Vault Data (`vd`) Contract

Informative. (Folded from proposal 0005; machine form: `spec/schemas/vault-data.schema.json`.)

`vd` is the decrypted runtime payload the reference consumers share — the object passed to `renderContextBlock(vd)` (injection, Section 9) and `resolveConsultationCore(vd, request)` (consultation, Section 10) by the content script, the harness, and the MCP server alike. It is a runtime convention, **not a wire artifact**: the normative artifacts remain the Section 4 fingerprint and Section 5 user configuration carried inside it.

| Key | Carries |
|---|---|
| `phaedo_fingerprint` | The Section 4 fingerprint document (including `consult_policies`, Section 10.5). |
| `phaedo_behavioral_signals` | Cross-session accumulated signals feeding the layers (producer-internal shape in v0.1). |
| `phaedo_linguistic_profile` | Surface/style features used by injection synthesis. No conversation content. |
| `phaedo_session_metrics` | Session counters / persona-strength inputs. No conversation content. |

All keys are optional. An object with no `phaedo_fingerprint` is a **legitimately empty vault** (a fresh user — nothing to render, not an error). A key present with the wrong type is a **contract breach** — a producer bug, distinct from emptiness.

Guard behavior at the consumer boundary (the reference implementation's policy, recommended for any consumer):

- **Fail loud in development:** name the violating field precisely (`validateVaultData` in `context-block.js`; `validateVaultShape` in the reference resolver).
- **Fail safe in production:** a breached payload skips injection (renders nothing, with a warning) or returns a structured error (`internal_error` naming the field) from consultation. A consumer MUST NOT resolve silently with partial data from a breached payload.
- Weird-but-conforming content (odd signal values, empty layers) is **not** a breach; consultation resolves it defensively (typically at low deference or as an `insufficient_signal` abstain).
